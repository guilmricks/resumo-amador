01

A plataforma

O que cada aba realmente faz

A interface se divide em duas camadas: onde você pensa (Home) e onde você constrói de verdade (Code). Em volta disso, cinco recursos cobrem organização, execução autônoma e continuidade — a maioria das pessoas usa só o chat e nunca toca nos outros seis.

Núcleo
HomeChat e Cowork combinados — onde você refina ideia, delega tarefa de conhecimento (pesquisa, documento, planilha) e itera rápido sem compromisso de arquivo.
CodeAmbiente de programação real — terminal, sistema de arquivos, git, testes. É onde a ideia madura vira produto de verdade.
Organização e continuidade
ProjectsEspaço persistente por tema/cliente — agrupa conversas com arquivos e instruções fixas, sem reexplicar contexto toda vez.
ArtifactsOnde ficam os documentos/apps visuais publicados — histórico navegável, pronto pra compartilhar link.
CustomizeDiretório único de Skills, Connectors e Plugins — onde você instala e gerencia tudo que estende o Claude no chat e no Cowork.
Execução autônoma
ScheduledTarefas recorrentes. Rodam na nuvem (mesmo com PC desligado) se não usarem pasta local; rodam localmente (PC precisa estar ligado) se apontarem pra uma pasta no seu computador.
DispatchControle remoto do computador a partir do celular — usa arquivos locais, conectores e, com Computer Use, opera aplicativos abertos de verdade. Precisa do computador ligado.

Cada item, em profundidade

Home. É o ponto de entrada padrão — pesquisa, redação, análise, brainstorming. Também é onde o Cowork entra: você pode pedir uma tarefa mais longa ("levanta os pontos-chave desses 4 documentos e monta um resumo comparativo") e o Claude executa em segundo plano enquanto você continua conversando ou até fecha a aba.
Exemplo genérico: pedir uma análise comparativa de três fornecedores a partir de propostas em PDF, sem precisar abrir cada arquivo manualmente.

Code. Não é só "onde o código roda" — é onde o Claude tem acesso real ao sistema de arquivos: lê, edita, executa comandos, roda testes, versiona com git. A diferença central pra Home é que aqui existe estado persistente em disco, não só na conversa.
Exemplo genérico: construir uma ferramenta interna que processa uma planilha toda semana e gera um relatório automaticamente.

Projects. Resolve o problema de "reexplicar tudo de novo" — você fixa instruções (até ~8.000 caracteres, bem mais espaço que a instrução de perfil geral) e uma base de arquivos de referência (até 20 arquivos, 30MB cada, nos planos pagos), e toda conversa nova dentro daquele projeto já nasce com esse contexto carregado.
Detalhe que pega muita gente: conversas dentro do mesmo projeto não compartilham histórico entre si — só a base de arquivos é compartilhada. Cada conversa nova começa do zero na parte de diálogo, só herdando instrução e arquivos.
Exemplo genérico: um projeto por cliente ou por contrato, com escopo, glossário de termos e exemplos de output anteriores upados — qualidade da base de arquivos importa mais que quantidade.

Artifacts. Ao criar um novo, o Claude mostra 7 categorias de ponto de partida — cada uma direciona o tipo de resultado que ele prioriza, melhorando a qualidade da primeira tentativa:

Apps and websitesAplicações interativas com estado (formulário, calculadora, painel funcional).
Documents and templatesTexto longo estruturado — proposta, relatório, modelo reutilizável.
GamesLoops baseados em Canvas — mais lúdico, mas útil pra treinamento gamificado.
Productivity toolsDashboards e rastreadores — orçamento, cronograma, controle de progresso.
Creative projectsConteúdo visual — infográfico, diagrama, conceito ilustrado.
Quiz or surveyFluxo estruturado de pergunta e resposta, com pontuação — avaliação, formulário de admissão.
Start from scratchSem viés — o resultado depende inteiramente do quanto seu prompt for explícito sobre o formato.

O painel do artifact tem duas abas: Preview (roda ao vivo) e Code (mostra o código-fonte completo). Publicar gera um link público, atualizável sem trocar a URL.
Limitações reais: o Preview roda JavaScript de navegador, não um servidor — não dá pra chamar a maioria das APIs externas de dentro dele. "Baixar como HTML" entrega o código-fonte bruto do Claude, não um arquivo otimizado pra produção.
Exemplo genérico: publicar uma calculadora de orçamento simples pra um colega reutilizar, sem precisar instalar nada.

Existem dois "Artifacts" diferentes: o clássico, criado em Home, é público assim que publicado (qualquer pessoa com o link acessa, sem login). O do Claude Code, criado dentro de uma sessão de código, é privado por padrão — só o criador acessa, a menos que compartilhe manualmente dentro da organização. São mecanismos parecidos, mas com regra de visibilidade oposta por padrão — vale conferir qual dos dois você está usando antes de assumir que algo é público ou privado.

Customize. É o diretório único de 3 abas — Skills, Connectors e Plugins — de onde você estende o Claude no chat e no Cowork:

SkillsArquivos de instrução reutilizável. Claude lê só o nome/descrição de cada skill até uma bater com sua tarefa, depois carrega o resto — por isso ter dezenas instaladas não pesa no contexto.
ConnectorsLigação via MCP com serviço externo (Google Drive, Gmail, Slack etc.). É a "tomada" que dá ao Claude acesso a dado real fora da conversa.
PluginsEmpacota skills + connectors + comandos + subagentes num pacote só, instalado de uma vez.

Não confunda com as skills do Claude Code: as 7 skills que você instalou via terminal (Impeccable, Superpowers etc.) vivem num sistema separado, específico do Claude Code — instalado por linha de comando, não por aqui. O Customize > Skills do chat/Cowork é outro diretório, com sua própria lista (por padrão só vem com skill-creator, da própria Anthropic). Os dois coexistem, mas não se misturam automaticamente.

Onde fica o tom/instrução (não é aqui): personalização de tom não mora no Customize — fica em dois lugares diferentes: instrução de perfil em Settings, e Style por conversa no menu "+" da caixa de mensagem. Veja a tabela abaixo.

Personalização de tom — separada do Customize

São 3 camadas que se empilham, em lugares diferentes da interface:

Instruções de perfilSettings → General/Account. Vale em toda conversa, de qualquer projeto (limite ~1.500 caracteres). "Quem eu sou, como eu trabalho."
StylesMenu "+" da caixa de mensagem, em qualquer conversa. Tom por conversa — presets prontos (Normal, Learning, Concise, Explanatory, Formal) ou um estilo próprio, criado a partir de um exemplo de escrita sua.
Instruções de projetoDentro de cada Project. Vale só ali (limite ~8.000 caracteres) — mais espaço pra contexto de domínio.

As três rodam juntas em cada resposta — instrução de projeto + instrução de perfil + style escolhido, empilhadas. Regra prática: perfil pra identidade e tom permanente, projeto pra conhecimento de domínio, style pra ajuste pontual de uma conversa específica.
Exemplo genérico: instrução de perfil fixando tom direto e sem rodeio; instrução de projeto com o glossário técnico de um cliente específico; style "Formal" ligado só na conversa em que você está redigindo algo pra enviar externamente.

Scheduled. Transforma um pedido pontual em rotina. Você descreve a tarefa e a frequência uma única vez; a partir daí ela roda sozinha.
O detalhe que muda tudo: ao criar a tarefa, existe um campo de pasta. Deixe vazio (ou aponte pra um Project) e ela roda na nuvem, sobrevivendo ao PC desligado. Aponte pra uma pasta local do seu computador e ela passa a rodar só localmente — mesma restrição do Dispatch, precisa do PC ligado e acordado.
Exemplo genérico: um resumo de e-mail/calendário toda manhã roda liso na nuvem; uma tarefa que organiza arquivos numa pasta específica do computador exige o PC ligado, porque precisa tocar nesses arquivos de verdade.

Dispatch. É a ponte entre celular e computador — você manda uma instrução de qualquer lugar, e ela é executada no seu computador local, usando os arquivos e programas que já estão lá. Precisa do computador ligado e do app aberto, porque a execução acontece localmente, não na nuvem.
Exemplo genérico: estando fora, pedir pelo celular pra comparar três propostas em PDF salvas no computador — o resultado já está pronto quando você volta a se sentar.

Quando usar cada um

Uma pergunta simples resolve a maior parte das dúvidas: o resultado precisa virar arquivo/código que persiste, ou é uma resposta/documento que se resolve numa conversa? Primeiro caso → Code. Segundo caso → Home.

Diferença que importa: Dispatch é sempre local — por definição, é uma ponte pro seu computador específico. Scheduled pode ser local ou em nuvem, dependendo só de você apontar (ou não) pra uma pasta do computador na hora de criar a tarefa. Se a tarefa não precisa de arquivo local, prefira sempre deixar Scheduled sem pasta — roda mais confiável, sem depender do PC ligado.

02

Metodologia

Como o Superpowers substitui o preenchimento manual

A maior mudança de postura é parar de escrever prompt detalhado toda vez e passar a confiar num processo que já pergunta, planeja e verifica sozinho.

Antes e depois

Sem metodologiaVocê escreve um prompt longo tentando prever tudo, o Claude codifica direto, e só descobre o que faltou quando já está pronto (e errado).
Com SuperpowersVocê descreve a ideia solta. O Claude pergunta o que falta esclarecer, propõe um plano com critério de sucesso verificável, e só codifica depois de você aprovar.

O ciclo completo

  1. Entrevista — a skill de brainstorming pergunta o que precisa ficar claro, com base no que você descreveu.
  2. Plano — a partir das respostas, gera um documento de implementação com critérios de sucesso verificáveis (o que viraria um PROMPT.md se você escrevesse na mão).
  3. Execução com TDD — escreve teste que falha, código mínimo pra passar, depois refatora. Nunca marca algo como pronto sem o teste confirmar.
  4. Revisão — sub-skills dedicadas de code review, tanto pra pedir quanto pra receber revisão estruturada, antes de considerar terminado.

Referência completa — 13 sub-skills

O ciclo de 4 passos acima é a versão resumida. Por baixo, são 13 sub-skills especializadas — instaladas todas juntas, ativadas conforme a situação pede.

O ponto central: você deixou de ser quem escreve especificação técnica e virou quem aprova o plano que o Claude gerou. Isso é o que separa "operador digitando prompt" de "arquiteto revisando decisões".

03

Arquitetura de loops

Deixar de acompanhar prompt a prompt

Sessões longas degradam — a qualidade cai depois de muito contexto acumulado (a chamada "zona burra" da comunidade, geralmente depois de 100-150k tokens). Um loop reseta o contexto a cada iteração, mantendo o progresso em arquivos e no histórico do git em vez de na conversa.

Os 3 comandos nativos

/goalVocê dá a condição final; o Claude persegue até um avaliador confirmar que bateu. O mais útil no dia a dia.
/loopRepete um prompt em intervalo fixo, até você apertar Esc. Bom pra monitoramento.
/batchEspalha uma mudança mecânica grande em paralelo, com múltiplos agentes em worktrees separados.
Prompt — exemplo de /goal 
/goal [critério de sucesso verificável, ex: todos os testes passam e o lint está limpo]

Conexão com o capítulo 02: /batch na prática aciona duas sub-skills do Superpowers ao mesmo tempo — using-git-worktrees (isola cada agente numa cópia separada do repositório) e dispatching-parallel-agents (coordena o trabalho entre eles). Você não precisa chamar as duas manualmente; /batch já orquestra isso.

O que sustenta o loop por baixo

Independente do comando, o mecanismo é sempre o mesmo esqueleto de arquivos funcionando como memória externa, já que o contexto da conversa é descartado a cada iteração:

ObjetivoO que precisa existir ao final, com critério de sucesso verificável — não descreve como fazer.
Lista de tarefasPassos concretos, cada um só marcado como feito depois do critério confirmado — nunca por "parecer" pronto.
Comandos de build/testeO que o agente roda pra validar cada passo — sem isso, ele não tem como verificar nada.

Risco a evitar: loop bruto via terminal com --dangerously-skip-permissions dá acesso irrestrito de shell — há relatos reais de perda de pasta inteira sem isolamento. Use apenas dentro de ambientes de contêiner confiáveis, nunca direto na pasta principal do projeto.

04

Comandos úteis

O painel de controle de uma sessão — não só conversa

Além dos comandos de loop, existe um conjunto de comandos que controlam o que está acontecendo agora dentro da sessão — gestão de contexto, custo, navegação de histórico e qualidade. Digitar / sozinho mostra tudo que sua instalação atual expõe.

Gestão de contexto — o par mais usado no dia a dia
/compactResume o histórico da conversa, preservando o essencial. Aceita instrução do que reter — use quando o contexto passar de ~80%.
/clearReset completo — zera todo o histórico. Use ao trocar pra uma tarefa totalmente diferente, não relacionada à anterior.
/contextMostra quanto da janela de contexto está em uso agora — checagem rápida antes de decidir entre compact ou clear.
Navegação e histórico
/resumeRetoma uma sessão anterior — sem argumento, mostra um seletor das mais recentes.
/rewindReverte trabalho feito — permite escolher entre desfazer só código, só conversa, ou os dois. Atalho: Esc duas vezes.
/forkCria uma ramificação separada da sessão pra testar algo arriscado sem comprometer a principal.
Custo e performance
/costMostra quanto foi gasto na sessão (contas de API).
/usageMostra quanto resta do seu limite de uso (planos Pro/Max).
/modelTroca de modelo no meio da sessão — um modelo mais leve pra exploração, um mais robusto só na parte difícil.
/effortAjusta a profundidade de raciocínio — baixo pra tarefa repetitiva, alto pra debugging complexo ou decisão de arquitetura.
Planejamento e qualidade
/planForça modo de planejamento antes de tocar em arquivo crítico — pensa antes de editar.
/diffAbre um visualizador interativo de todas as mudanças da sessão — bom hábito antes de qualquer commit.
/code-reviewRevisão de código dedicada, separada da tarefa de construção.
/security-reviewRevisão focada em vulnerabilidades — injeção, autenticação, exposição de dado.
Paralelismo
/backgroundManda uma tarefa demorada (teste, build) rodar em segundo plano, sem travar a sessão.
Configuração e manutenção
/initCria o CLAUDE.md inicial do projeto — memória persistente entre sessões.
/permissionsDefine o que o agente pode fazer sem pedir confirmação.
/memoryEdita a memória salva do projeto diretamente.
/doctorDiagnostica a saúde da instalação — bom primeiro passo quando algo se comporta estranho.
/checkupAuditoria completa da configuração: limpa skill/plugin não usado, atualiza a versão, organiza CLAUDE.md volumoso em arquivos menores. Confirma cada mudança antes de aplicar.
/insightsAnalisa seu próprio histórico de uso e aponta prompts repetidos que valeriam virar comando ou skill.

Atalhos de teclado (fora do /)

Shift+TabAlterna entre modo normal, auto-aceitar edições, e modo de planejamento.
! (prefixo)Roda um comando bash direto, sem o Claude interpretar — mais rápido, gasta menos token. Ex: !git status.
@ (prefixo)Autocompleta caminho de arquivo — mais rápido que descrever o arquivo por extenso.

Sequência recomendada numa sessão longa

Checagem de saúde + limpeza antes de continuar 
/context
/compact mantenha só o essencial da tarefa atual
/diff
/cost

Rode /context periodicamente durante sessões longas — um loop autônomo pode consumir contexto e custo mais rápido do que parece, especialmente combinado com /goal.

Fluxo completo

Do zero ao mantido — juntando plataforma, metodologia, loops e comandos

Este é o fio que amarra tudo: cada etapa usa uma aba específica da plataforma e uma combinação específica de skills/comandos.

1

Ideia

Descreva o que quer construir em linguagem natural, sem estrutura técnica. A entrevista de esclarecimento acontece sozinha.

Manutenção

Rotina recorrente vai pra Scheduled (roda mesmo com o PC desligado). Ajuste pontual do celular usa Dispatch (precisa do PC ligado). claude-mem mantém o contexto do projeto vivo entre sessões. Task Observer acumula padrão pra virar skill nova.

06

Skills

Instalação e uso completos

Sete skills cobrindo design, rigor técnico, descoberta, memória, compressão de tokens e escrita. Cada uma resolve um problema específico — nenhuma se sobrepõe de forma redundante às outras.

Escopo: as skills abaixo são do Claude Code, instaladas por linha de comando. É um sistema separado do Customize > Skills do chat/Cowork (esse último vem só com skill-creator por padrão, e se instala pela interface, não pelo terminal). Veja a seção 01 pra entender a diferença. Todo comando que começa com / (como /plugin ou /impeccable init) é digitado dentro de uma sessão aberta do Claude Code — não no chat do navegador nem direto no PowerShell.

Impeccable

Design de interface · por projeto

Skill de design front-end que evita a estética genérica de IA (gradientes roxo-azul, cards dentro de cards, mesma fonte de sempre). Exige contexto de marca antes de gerar qualquer interface e usa paletas de cor e escalas tipográficas avançadas.

Instalar

  1. Instala uma vez, de forma global — vale para todos os projetos.
  2. Rode o comando no PowerShell.
  3. Depois, dentro de cada projeto visual, rode /impeccable init uma única vez para definir o contexto de marca daquele projeto.
PowerShell 
npx impeccable install
Dentro do Claude Code, na pasta do projeto 
/impeccable init

Usar

A skill entra sozinha quando você pede algo visual. Exemplos de prompts:

Prompt — tela nova 
cria a tela de [nome da tela], estilo [referência visual]
Prompt — dashboard 
redesenha o dashboard do [projeto], deixando mais profissional

Superpowers

Metodologia de engenharia · global

Framework completo de disciplina de desenvolvimento: antes de codar, esclarece requisitos, monta um plano de implementação e força TDD real (teste que falha → código mínimo → teste passa → refatora). Instala 13 sub-skills de uma vez.

Instalar

  1. Abra o Claude Code (claude) em qualquer pasta.
  2. Rode o comando de adicionar o marketplace — espere confirmar antes do próximo passo.
  3. Rode o comando de instalação do plugin.
  4. Feche e reabra o Claude Code para ativar.
Dentro do Claude Code — passo 1 
/plugin marketplace add obra/superpowers-marketplace
Dentro do Claude Code — passo 2 
/plugin install superpowers@superpowers-marketplace --global

Usar

Não precisa chamar por comando — entra sozinha quando você pede para construir algo. Exemplo de prompt:

Prompt — tarefa técnica com rigor 
implementa a função de [tarefa técnica], com validação de [critério de qualidade]
Prompt — pular a etapa de perguntas 
sem brainstorm, só implementa direto: [descreva a tarefa aqui]
Verificar instalação 
/skills

07

Lista de observação

Avaliadas · não instaladas agora

Ferramentas analisadas que fazem sentido apenas em cenários específicos de escala — não valem a pena no volume de trabalho de um desenvolvedor solo com poucos projetos por vez. Ficam registradas aqui com o gatilho que justificaria reconsiderar cada uma.

Ruflo (antigo Claude Flow)

Plataforma de orquestração multiagente ("hive-mind") construída sobre o Claude Code — coordena múltiplos agentes especializados em paralelo (pesquisador, tester, reviewer etc.) com memória compartilhada entre eles.

Por que não agora: configurações multiagente ingênuas costumam gastar mais tokens do que uma sessão única bem estruturada, porque cada agente paga overhead de contexto próprio. Pensada para times enterprise, não builds sequenciais de um projeto por vez.

Reconsiderar quando: houver várias frentes de trabalho genuinamente paralelas e interdependentes rodando ao mesmo tempo — não apenas vários projetos separados.

↗ ruvnet/claude-flow no GitHub

Graphify

Constrói um grafo de conhecimento do código antes do Claude tocar em qualquer arquivo — mapeia importações, chamadas de função e módulos centrais.

Por que não agora: em repositórios com menos de 30 arquivos, o custo de construir o grafo supera a economia de tokens. Desenhada para codebases grandes (500+ arquivos), com dependência de mantenedor único.

Reconsiderar quando: algum projeto crescer significativamente em volume de arquivos e módulos interligados.

↗ Graphify-Labs/graphify no GitHub

08

Segurança — checklist consolidado

Os riscos reais espalhados por esta apostila, num só lugar

  • !
    Loop via terminal

    --dangerously-skip-permissions dá acesso irrestrito de shell — só usar dentro de ambiente de contêiner confiável, nunca na pasta principal. Detalhe na seção 03.

  • !
    claude-mem

    API HTTP local sem autenticação (porta 37777) — qualquer processo na máquina pode ler as observações. Risco baixo em uso pessoal, evitar em máquina compartilhada.

  • !
    Dispatch + Computer Use

    Claude pode operar aplicativos abertos de verdade — começar sempre em modo leitura (buscar, resumir, comparar) e exigir aprovação antes de qualquer ação que envie, apague ou modifique algo importante.

  • !
    Skills de terceiros

    Rodam com permissões amplas do agente. Revisar o que a skill faz antes de instalar, especialmente de fontes menos conhecidas.

  • !
    Loop sem gate de teste

    Sem verificação automática, o agente pode marcar tarefa como "pronta" quando só parece pronta. Critério de sucesso sempre verificável, nunca subjetivo.

  • !
    Artifacts publicados

    Um artifact clássico (Home) publicado fica acessível a qualquer pessoa com o link, sem login — nunca publique algo com dado sensível de cliente ou empresa. O artifact do Claude Code já nasce privado, mas confirme antes de assumir isso.

09

Glossário

Termos usados ao longo da apostila

API
Interface que permite um programa "conversar" com outro. "Chamar uma API" é pedir dado ou ação a um serviço externo.
Artifact
Conteúdo autocontido (documento, app, diagrama) que o Claude gera num painel separado da conversa, pronto pra visualizar, editar e publicar. Existem duas versões com regras diferentes: a clássica (Home, pública ao publicar) e a do Claude Code (privada por padrão).
CLAUDE.md
Arquivo de instrução e memória que o Claude Code lê automaticamente no início de cada sessão, dentro de um projeto.
Computer Use
Capacidade do Claude de operar aplicativos abertos no computador como faria uma pessoa — clicar, digitar, navegar entre telas.
Conector (MCP)
Ligação configurada entre o Claude e um serviço externo (e-mail, calendário, planilha na nuvem etc.) através do protocolo MCP, dando acesso a dados e ações daquele serviço dentro da conversa.
Cowork
Modo do Claude que executa tarefas de conhecimento (pesquisa, documento, planilha) em segundo plano, entregando o resultado pronto.
Deploy
Colocar uma aplicação no ar, acessível por link ou domínio próprio, fora do ambiente de desenvolvimento.
Effort
Nível de profundidade de raciocínio aplicado numa tarefa. Baixo economiza tempo e custo em tarefas simples; alto vale a pena em decisões complexas ou debugging difícil.
Fork (de sessão)
Ramificação separada de uma conversa/sessão, usada pra testar algo arriscado sem comprometer a principal.
Gate de verificação
Condição objetiva (teste passando, comando sem erro) que precisa ser confirmada antes de uma tarefa ser considerada completa. No Superpowers, é a sub-skill verification-before-completion que aplica essa checagem.
Grafo de conhecimento (de código)
Mapa das relações entre arquivos e funções de um projeto, usado pra navegação mais eficiente em vez de busca arquivo por arquivo.
Hive-mind
Arquitetura em que múltiplos agentes de IA são coordenados como uma colônia, com papéis diferentes — planejador, executor, revisor.
Janela de contexto
Quantidade de conversa/código que o modelo consegue "ver" de uma vez. Enche com o tempo e degrada a qualidade da resposta perto do limite.
Loop / arquitetura de loop
Técnica de deixar o Claude iterar sozinho até um critério de sucesso ser verificado, sem você acompanhar prompt a prompt.
MCP (Model Context Protocol)
Protocolo aberto que permite ao Claude se conectar a ferramentas e serviços externos de forma padronizada — é o que torna um conector possível.
npm / npx
Gerenciador de pacotes do Node.js. npx roda um pacote uma única vez, sem instalar permanentemente — é o comando mais comum pra instalar skill nova.
Plugin / Marketplace
Um plugin empacota várias skills/comandos juntos; o marketplace é o repositório de onde ele é instalado.
Prompt
A instrução ou pedido que você escreve pro Claude — desde uma pergunta simples até um plano detalhado de várias linhas.
Sandbox / contêiner isolado
Ambiente separado do seu sistema principal (Docker, devcontainer) onde uma ação arriscada não afeta seus arquivos reais.
Skill
Pacote de instrução/comportamento reutilizável que o Claude carrega — pode ser da comunidade ou criado por você.
Style
Predefinição de tom aplicada a uma conversa específica (formal, conciso, didático etc.), sem alterar as instruções permanentes de perfil ou de projeto.
Subagente
Sessão auxiliar que o Claude Code invoca pra uma tarefa específica, mantendo a conversa principal limpa.
TDD (Test-Driven Development)
Escrever o teste antes do código: teste falha → código mínimo pra passar → refatora. Garante que "pronto" seja verificável, não uma impressão.
Token
Unidade de texto que o modelo processa, aproximadamente um pedaço de palavra. Custo e limite de contexto são medidos em tokens.
Worktree
Cópia de trabalho paralela de um mesmo repositório git, usada pra rodar múltiplos agentes ou tarefas ao mesmo tempo sem que um interfira no outro. No Superpowers, é a sub-skill using-git-worktrees que gerencia isso.
Zona burra
Apelido da comunidade pro ponto em que a janela de contexto está tão cheia que a qualidade da resposta cai visivelmente.
